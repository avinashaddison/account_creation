We should switch the automation to use Browserless Stealth Chromium with Playwright instead of running a local browser on Replit.

Proposed setup:

1. Connect Playwright to Browserless Stealth Chromium:
   

import { chromium } from "playwright-core";

const browser = await chromium.connectOverCDP(
  "wss://production-sfo.browserless.io/chromium/stealth?token=YOUR_API_TOKEN"
);

// Create a new context with proper fingerprint
const context = await browser.newContext({
  locale: "en-US",
  timezoneId: "America/New_York",
  viewport: { width: 1366, height: 768 },
  userAgent:
    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 Chrome/120.0.0.0 Safari/537.36"
});

const page = await context.newPage();

// Warm up session (important for Gigya cookies)
await page.goto("https://www.la28.org", { waitUntil: "domcontentloaded" });

// Small delay to look human
await page.waitForTimeout(3000);

// Now go to login page
await page.goto("https://id.la28.org");

// Live browser session (optional debugging)
const cdpSession = await context.newCDPSession(page);
const { liveURL } = await cdpSession.send("Browserless.liveURL");

console.log("Live browser session:", liveURL);

// Wait for manual completion if needed
await new Promise((resolve) =>
  cdpSession.on("Browserless.liveComplete", resolve)
);

// Continue automation
await browser.close();



2. Create a proper browser context to match a real US user:

   * locale: en-US
   * timezoneId: America/New_York
   * viewport: 1366x768
   * realistic Chrome user-agent

3. Warm up the session before login:

   * open https://www.la28.org
   * wait a few seconds
   * then navigate to https://id.la28.org

4. Perform the Gigya login through the browser UI instead of calling the Gigya REST API directly.

5. Add small human-like delays and interactions so the login flow looks like normal browser activity.

6. Route the Browserless session through a US residential proxy if possible to avoid IP reputation issues.

This setup should be more stable for Gigya authentication and avoid the 451002 errors we were seeing with Replit datacenter IPs and direct API requests.
